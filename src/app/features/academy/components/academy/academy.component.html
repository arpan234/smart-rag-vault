RAG 101: What, Why, When & How?

Welcome to the laboratory! If you are new to Retrieval-Augmented Generation, start with these four fundamental concepts.

1. What is RAG?

Retrieval-Augmented Generation is a way to link pre-trained AI brains with your own private text files (PDFs, text files, SQL databases) in real time.

The Definition
2. Why do we need it?

AI brains normally freeze in time after training and make up answers (hallucinate). RAG provides a search engine that grabs facts first, forcing 100% accuracy.

The Benefit
3. When to use it?

Choose RAG when documents change frequently or when facts are private. Choose Fine-Tuning when you want to change the model's behavior, tone, or dialect.

RAG vs. Fine-Tuning
4. How does it work?

Convert files to vector coordinates ➔ Match query coordinate in Vector DB ➔ Combine query + fetched context ➔ Feed prompt to LLM to write answer.

The Workflow

What does RAG mean?

Retrieval (R) Searching through a database of uploaded documents to find segments semantically related to the user's question.
Augmented (A) Injecting the retrieved database text blocks into the LLM prompt, backing the request with concrete contextual evidence.
Generation (G) The LLM synthesizes a fluent, natural response based on the injected context, inserting inline references and source citations.

Stage Description @if (animStep() === 1) {

1. User Inputs Prompt

The pipeline begins when a user asks a question about details not present in the LLM's public pre-training memory (e.g. secret keys or local files). The prompt must be processed dynamically before being sent to the AI.

Input is captured in the client interface.
} @else if (animStep() === 2) {

2. Text converted to Vector

Computers cannot calculate mathematical similarities on raw text strings. The system calls an **embedding model** (such as Gemini's `text-embedding-004`) to map the text query into a high-dimensional vector coordinate that captures its exact semantic meaning.

The coordinate encodes topics like food, server, or mascot.
} @else if (animStep() === 3) {

3. Vector Database Scanning

The database performs a math search. It takes the query coordinate and runs a **Cosine Similarity** check against all pre-stored text chunk coordinates in the vault. The segment that has the closest matching angle is ranked highest.

Comparing coordinate directions instead of literal words.
} @else if (animStep() === 4) {

4. Fetching Relevant Chunks

The database returns the text of the highest-matching document segments (chunks). The file context has now been successfully isolated, extracting only the few sentences that contain the actual facts required to answer the question.

Found matching document chunk with high cosine score.
} @else if (animStep() === 5) {

5. Merging & Augmenting Prompt

The application wraps the user's question and the retrieved text context inside a structured system template. This process is called **Augmentation**. It orders the LLM to only answer the user's question by citing the retrieved facts.

Injects instructions: "Answer ONLY using this reference."
} @else if (animStep() === 6) {

6. LLM Generates Fact-Backed Answer

The augmented prompt is passed to the LLM. Because the exact answer is right there in the context, the model does not need to guess. It reads the guidelines, compiles a factual response, and streams it back.

Answer is fully grounded in the retrieved database context.
}

Step-by-Step Math Breakdown

1. Dot Product (A • B) Value = {{ vectorAnalysis().dotProduct }}

Multiplying aligned words together: "algorithms" (1*1) + "vectors" (1*1) + ...

2. Vector Magnitudes (Lengths) ||A|| = {{ vectorAnalysis().magnitudeA }} | ||B|| = {{ vectorAnalysis().magnitudeB }}

Represents text length density. Computes square root of sum of squared word occurrences.

3. Cosine Calculation Synthesis
cos(θ) = (A • B) / (||A|| * ||B||)
cos(θ) = {{ vectorAnalysis().dotProduct }} / ({{ vectorAnalysis().magnitudeA }} * {{ vectorAnalysis().magnitudeB }})
cos(θ) = {{ vectorAnalysis().similarity }}

By dividing the dot product by the product of their length magnitudes, we normalize the score. This ensures sentence length difference doesn't distort similarity matching.
